# AETHER 더빙 QC 파이프라인 재설계 — 설계 문서

- 날짜: 2026-07-14
- 상태: 사용자 승인 완료 (섹션별 승인)
- 대상 저장소: dubbing_qc

## 1. 배경과 문제 정의

한국 영화의 영어 AI 더빙 결과물의 로컬화 품질(직역 오류, 문화적 표현 실패)을
평가하는 모델·도구가 시중에 없다. 사내 검수자가 실무에 쓸 QC 도구를 만든다.

**현재 코드의 구조적 문제:** 기존 파이프라인은 한국어 원문을 스스로 번역한 뒤
자기 번역을 검수한다. 진짜 QC는 외부에서 제작된 **더빙 완성본**의 영어 음성을
STT로 추출해 원본과 대조하는 것이다. 이 방향 전환이 이번 재설계의 핵심이다.

**입력:** 한국어 원본 영상 + 영어 더빙 완성본 영상 (둘 다 항상 제공됨)
**출력:** ① 타임코드별 수정 지시서 ② 통과/조건부 통과/반려 판정 ③ 대시보드 열람
**규모:** 장편 영화 전체(~2시간) 배치 처리
**보안 제약:** 외부 AI API로의 영상/음성/대본 전송 가능 여부 미확정 →
프로바이더 추상화로 대비 (아래 §4)

## 2. 전략: 단계적 진화 (LLM Judge → 하이브리드 → 로컬)

| 단계 | 시점 | 구성 |
|---|---|---|
| 1단계 (지금 구현) | 즉시 | Gemini 멀티모달 페르소나 패널 + 지식베이스 프롬프트 + 결정론적 룰 체크 + 피드백 축적 루프 |
| 2단계 (데이터 ~10편 축적 후) | 수개월 뒤 | xCOMET(또는 MetricX-25-QE) 점수 스크리너를 패널 앞단에 추가 — 전 세그먼트 스크리닝 후 저점수만 LLM 정밀 분석 (비용 절감). 아카이브 최종 SRT로 합성 부정 예시를 만들어 부트스트랩 가능 |
| 3단계 (보안 정책이 요구할 때) | 미정 | LoRA 튜닝한 로컬 LLM(Qwen3/EXAONE)이 클라우드 Judge를 교체 |

핵심 설계 수: **검수자의 승인/반려/수정 행위가 자동으로 (초안 + 지적 + 최종)
학습 데이터로 축적**되어, 도구를 쓸수록 다음 단계 모델이 준비된다.
학습은 실시간이 아니라 주기적 배치 재학습이며, 골든 평가 세트로 검증 후 교체한다.

이번 구현 범위는 **1단계 전체 + 2·3단계를 위한 인터페이스 자리만** 확보하는 것이다.

## 3. 파이프라인 (데이터 흐름)

```
[원본 영상]                     [더빙 완성본]
    │ ffmpeg 오디오 추출             │ ffmpeg 오디오 추출
    ▼                               ▼
[한국어 음성] ── STT ──▶ 한국어 세그먼트
[영어 음성]  ── STT ──▶ 영어 더빙 세그먼트
               │
               ▼
① 타임코드 정렬: 한↔영 세그먼트 짝 맞추기 (AlignedPair 생성)
② 결정론적 룰 체크 (LLM 없음): 번역 누락 / 발화속도 초과 /
   싱크 오버플로 / 정렬 신뢰도 저하 / 음질(클리핑·SNR·드롭아웃)
③ LLM Judge 페르소나 패널 (씬 청크 단위, 3개 페르소나 독립 평가 후 병합)
④ 판정 엔진: 축별 MOS 산출 → 통과/조건부/반려
⑤ 대시보드 + 수정 지시서 내보내기 (검수자 승인/반려/수정)
⑥ 피드백 저장소: 검수 행위를 JSONL로 축적 (미래 학습 데이터)
```

- 기존 `/translate` 단계는 **삭제** — "더빙본 STT + 정렬"이 대체한다.
- 룰 체크는 기계적으로 확실한 것만 담당하는 저렴한 거름망이며,
  QC의 본체(직역/문화/톤 평가)는 ③ 패널이다.
- 2시간 영화는 씬 경계 기준 청크로 분할해 배치 처리하고 진행률을 노출한다.
- xCOMET 스크리너는 ③ 앞에 인터페이스 자리만 두고 이번에는 구현하지 않는다.

### ③ 페르소나 패널 상세

동일 씬 청크를 3개 페르소나가 독립 평가한다:

| 페르소나 | 관점 | 입력 |
|---|---|---|
| 한국 문화·언어 전문가 | 호칭/관용구/존댓말 뉘앙스 보존 여부, 직역 훼손 | 텍스트 쌍 + 지식베이스 |
| 영어 원어민 일반 시청자 | 한국어를 모르는 시청자에게 자연스러운지, 번역투 여부 | 영어 텍스트 중심 |
| 더빙 연출가 | 캐릭터 톤 일치, 씬 정서, 대사 호흡, 억양 | 텍스트 쌍 + **오디오 클립** (Gemini 네이티브 오디오 입력) |

**병합기(Merger) 규칙:**
- 2개 이상 페르소나가 같은 세그먼트 지적 → 심각도·신뢰도 상향
- 1개만 지적 → 신뢰도 하향해 유지 (검수자가 최종 판단)
- 수정안이 서로 다르면 → `alternatives`로 나란히 제시, 검수자가 선택
  (선택 기록 자체가 학습 데이터가 됨)

비용: 페르소나 3배 호출이지만 Gemini Flash 기준 영화 한 편 수 달러 수준.
오탐 검수 인건비 대비 패널의 정밀도 향상이 총비용에서 이득.

## 4. 백엔드 컴포넌트 구조

```
backend/app/
├── main.py                  # API 엔드포인트 (업로드/STT/QC실행/판정/피드백)
├── schemas.py               # 데이터 계약 (§5)
├── core/
│   ├── pipeline.py          # 총괄: ①정렬→②룰체크→③패널→④판정
│   ├── alignment.py         # 신규 ① 한↔영 타임코드 정렬
│   ├── rule_checks.py       # 신규 ② 결정론적 체크 (누락/발화속도/싱크/음질)
│   ├── judge_panel.py       # 신규 ③ 페르소나 3개 실행 + 병합기
│   ├── verdict.py           # 신규 ④ 축별 MOS + 판정
│   ├── context.py           # 유지   씬 맥락 추출 (패널 프롬프트 공급)
│   ├── voice_qc.py          # 유지   음색 일관성 검사
│   └── localization.py      # 흡수   rubric 프롬프트 → judge_panel로 이동
├── providers/               # 신규 ★ 모델 프로바이더 추상화
│   ├── base.py              # 계약: transcribe() / judge() (+ score() 자리만)
│   ├── gemini.py            # Gemini 구현
│   └── mock.py              # API 키 없을 때 폴백 (현행 mock 로직 이관)
├── knowledge/               # 신규 지식베이스 (YAML, 코드 수정 없이 편집 가능)
│   ├── honorifics.yaml      # 호칭 처리 규칙 (형/오빠/부장님…)
│   └── idioms.yaml          # 관용구 사전 (눈치/어이없네/밥…)
└── feedback/
    └── store.py             # ⑥ 승인/반려/수정 기록 → JSONL append
```

**프로바이더 계약 (2함수 + 예약 1함수):**
- `transcribe(audio, lang) → 세그먼트 목록`
- `judge(청크, 페르소나, 지식베이스, 오디오클립?) → QCFinding 목록`
- `score(세그먼트쌍) → 품질 점수` — 2단계 xCOMET용 예약, 이번엔 미구현

`judge_panel.py`는 계약만 알고 특정 모델을 모른다. 모델 교체 시 어댑터 코드는
작지만, 프롬프트 재조정과 품질 검증이 진짜 비용이므로 **골든 평가 세트**
(정답을 아는 검증용 영화 1~2편)를 운영 초기에 구축한다.

**피드백 저장소는 JSONL 파일.** append 전용이라 동시성 문제가 없고,
학습 데이터 변환이 즉시 가능하다. 조회 요구가 생기면 그때 DB로 임포트한다.

```jsonl
{"movie": "...", "segment_id": "...", "korean": "...", "dubbed": "...",
 "ai_findings": [...], "persona_suggestions": {...},
 "reviewer_action": "approved|rejected|modified", "final_text": "...",
 "timestamp": "..."}
```

## 5. 데이터 계약

**AlignedPair** (기존 단일 세그먼트의 original/translated 동거 구조를 대체):

```
AlignedPair
├── korean:  {start, end, speaker, text}   ← 원본 STT
├── dubbed:  {start, end, speaker, text}   ← 더빙본 STT
├── scene_id                                ← 씬 청크 소속
└── alignment_confidence                    ← 낮으면 그 자체가 finding
```

**QCFinding** — 기존 필드(severity/issue_type/description(한국어)/
recommendation(영어)/confidence) 유지 + 추가:
- `axis`: 사내 5축 중 소속 (음질/감정 표현/싱크 정확도/자연스러움/언어 적합성)
- `source`: `"rule"` | `"persona:문화전문가"` | `"persona:원어민"` | `"persona:연출가"`
- `agreement`: 동의한 페르소나 수 (1~3)
- `alternatives`: 페르소나별 수정안 목록

## 6. 판정 체계 — 사내 5축 MOS 기준 채택

사내 기존 평가 체계(5축 × MOS 1~5등급)를 공식 출력 형식으로 사용한다.
100점 감점제(high -15 / medium -8 / low -3)는 내부 계산용으로만 유지.

**축 ↔ 컴포넌트 매핑:**

| 사내 평가축 | 담당 |
|---|---|
| 언어 적합성 (번역 뉘앙스, 문화적 자연스러움) | 패널: 문화전문가 + 원어민 |
| 감정 표현 (감정선, 대사 톤) | 패널: 연출가 (오디오 입력) |
| 자연스러움 (억양, 말 흐름) | 패널: 원어민 + 연출가 |
| 싱크 정확도 (입 모양, 장면 전환 타이밍) | 룰 체크: 발화속도/싱크 오버플로 |
| 음질 (잡음, 왜곡, 클리핑, 명료도) | 룰 체크: 신호 분석 (librosa/ffmpeg) |

**산출:** finding을 axis별 집계 → 축별 감점 → 축별 MOS(1~5)

**판정 규칙:**
- ✅ 통과: 전 축 MOS ≥ 4
- ⚠️ 조건부 통과: 최저 축 MOS = 3 → 수정 권고 목록 첨부
- ❌ 반려: 어느 축이든 MOS ≤ 2, **또는 high finding 1건 이상** → 수정 지시서 필수

high 1건 즉시 반려인 이유: 치명적 오역 하나가 작품 평판을 망치는 도메인이라
평균으로 희석되면 안 된다. 감점→MOS 변환 기준과 판정 임계값은 설정 파일로 분리.

**캘리브레이션:** 리포트가 기존 수기 평가서와 같은 5축 형식이므로 사람 MOS와
AI MOS를 나란히 비교해 어긋나는 축의 프롬프트를 조정한다.

## 7. 대시보드 (프론트엔드)

기존 자산(비디오+파형 연동, finding 카드, 다크테마) 유지. 3개 뷰:

1. **프로젝트 뷰 (신규):** 원본+더빙본 업로드, 파이프라인 진행률
   (STT → 정렬 → 룰체크 → 패널 씬 n/m 분석 중)
2. **검수 뷰 (기존 개조):** 좌측 더빙본 비디오+파형(finding 클릭 시 구간 점프),
   한↔영 대사 대조. 우측 finding 카드(심각도/축/씬 필터), 카드 상세에
   지적 사유(한국어) + 페르소나별 수정안 비교 + [승인][반려][직접수정] 버튼
3. **판정/리포트 뷰 (신규):** 5축 MOS 스코어카드 + 판정 배너,
   검수자 확정 finding만 모은 수정 지시서, 엑셀/PDF 내보내기

**워크플로 규칙:**
- AI 판정은 가판정. 검수자가 finding을 승인/반려한 뒤 오탐이 제외된 상태로
  MOS·판정 재계산 → 확정. (AI는 보조, 검수자가 결재자)
- 모든 버튼 클릭이 피드백 저장소에 기록된다. "Apply AI Fix" 버튼은 제거.

## 8. 오류 처리

- STT 실패/저신뢰 구간: 해당 세그먼트를 `alignment_confidence` 저하 finding으로
  표면화하고 파이프라인은 계속 진행 (전체 중단 금지)
- LLM 응답 파싱 실패: 청크 단위 1회 재시도 후 실패 시 해당 청크를
  "미분석" 상태로 표시 — 검수 뷰에서 수동 재실행 가능
- API 키 부재: mock 프로바이더 폴백 (현행 동작 유지)
- 장시간 배치: 씬 청크 단위 체크포인트 저장 — 중단 시 이어서 재개

## 9. 테스트 전략

- 룰 체크·정렬·병합기·판정: 결정론적이므로 단위 테스트로 고정
- 패널 프롬프트: 골든 평가 세트(오류를 심어둔 짧은 샘플 클립)로 회귀 검증
- mock 프로바이더로 파이프라인 전 구간 E2E 테스트 (API 키 불필요)

## 10. 범위 밖 (명시적 제외)

- xCOMET/MetricX 스크리너 구현 (2단계 — 인터페이스 자리만)
- 로컬 LLM 서빙/LoRA 학습 (3단계)
- 피드백 저장소의 DB 전환
- 자동 재더빙/음성 합성 — 이 도구는 리포트·판정까지만
- 수정 지시서의 사내 양식 맞춤 — 구현 단계에서 샘플 수령 후 반영

## 11. 축적 데이터의 학습 활용 로드맵 (2·3단계 상세)

이번 구현 범위는 아니지만, 1단계 설계가 이 로드맵을 가능하게 하도록
피드백 저장소 형식을 고정하는 근거이므로 명시한다.

**학습 데이터 변환:** 피드백 JSONL 한 줄이 그대로 학습 예제가 된다.
- `reviewer_action: approved` → (한영 쌍, AI finding) = 정답 지적 예제
- `reviewer_action: rejected` → 오탐 예제 (지적하면 안 되는 케이스)
- `reviewer_action: modified` + `final_text` → (오류 세그먼트 → 올바른 수정안) 생성 예제
- 지적 없이 통과한 세그먼트 → "정상" 라벨 (점수 모델용 대량 양성 예제)

**필요 데이터량과 도달 시점** (편당 세그먼트 ~1,000–1,500개, 지적 50–200건 기준):

| 대상 모델 | 필요 데이터 | 필요 축적량 |
|---|---|---|
| 2단계: xCOMET 계열 점수 스크리너 튜닝 | 전 세그먼트의 정상/오류 라벨 | 약 10편 |
| 3단계: 로컬 LLM LoRA (지적+설명+수정안 생성) | 양질의 (오류→지적→수정) 3,000–5,000건 | 약 20–50편 |

**부트스트랩 (선택 가속):** 아카이브의 최종 SRT를 정답 삼아 직역투 번역을
합성 생성 → (나쁜 번역 vs 정답) 쌍으로 점수 모델을 사전 학습.
운영 데이터 축적 전에 2단계를 앞당길 수 있다.

**LoRA 학습 방식 (3단계):**
- 베이스 모델 후보: Qwen3 14B–32B(한영 다국어 최상위), EXAONE(한국어 특화),
  Llama 3.3, Gemma 3 — 데이터가 쌓인 시점의 최신 오픈웨이트로 결정 (지금 확정하지 않음)
- 학습 과제: judge 계약과 동일한 입출력 — (씬 청크 + 지식베이스) → QCFinding JSON.
  프로바이더 계약을 그대로 학습 포맷으로 쓰므로 교체 시 파이프라인 변경이 없다
- 학습은 실시간이 아닌 **주기적 배치 재학습** (분기/월 단위)
- 교체 기준: 골든 평가 세트에서 현행 클라우드 Judge와 성능 비교 후
  동등 이상일 때만 교체. 검증 없는 자동 교체 금지 (오염 데이터 방어)
- 서빙: 로컬 GPU + vLLM/Ollama — 보안 정책이 외부 전송 금지로 확정될 때
  구축하는 인프라 작업
